## R12 — Two Bug Fixes

### Bug 1: `main_function` column name mismatch in hierarchy endpoint

`GET /api/hierarchy` was referencing `main_function` but the actual column in `devices` table
is `main_function` — query was aliased correctly in some controllers but not hierarchy.

**Fix:** corrected the column reference so the full building → floor → rack → device tree
returns `main_function` values without SQL error.

### Bug 2: Camera endpoints blocked for `user` role

`GET /api/cameras` and `GET /api/cameras/{id}` were decorated with `[Authorize(Roles = "admin")]`
but the role matrix allows `user` role read access to cameras.

**Fix:** changed to `[Authorize(Roles = "admin,user")]` on both camera GET endpoints.

**Commit:** `1ed023b` fix(hierarchy,cameras): fix main_function column name + allow user role for GET cameras

## R15 — bcrypt Hash Mismatch Fixed

**Symptom:** `POST /api/auth/login` returned 401 for `admin` / `Admin@SSM1` (correct
credentials per schema comment at `SSM_schema_v2.sql:573`).

**Root cause:** The DB was seeded with a different bcrypt hash than what the schema file
showed. The plaintext password was correct; the hash in the DB was not.

**Fix:** Generated fresh `bcrypt(12)` hashes via Python and updated both rows in `SSM_DB`:

```sql
UPDATE users SET pw_hash = '<new_hash>' WHERE username = 'admin';
UPDATE users SET pw_hash = '<new_hash>' WHERE username = 'ssm_user';
```

Verified with `bcrypt.checkpw` — both return `True`.

| Username   | Password    |
|------------|-------------|
| `admin`    | `Admin@SSM1`|
| `ssm_user` | `User@SSM1` |

No code changes — `authController.cs` was correct throughout.

**Commit:** `465ce5b` fix(auth): reset bcrypt hashes
